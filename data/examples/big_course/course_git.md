# Course A: Git для локального контроля версий

## Описание курса

Курс посвящен практическому освоению Git как локальной системы контроля версий. Студенты учатся понимать историю проекта, безопасно фиксировать изменения, работать с ветками, решать конфликты и восстанавливать рабочее состояние без потери данных.

Курс рассчитан на студентов, которые уже умеют работать с файловой системой, терминалом и простыми текстовыми проектами. Все примеры выполняются локально: remote repositories, pull requests и GitHub Actions упоминаются только как контекст, но не являются основным предметом курса.

## Цели курса

- Сформировать понимание Git repository как истории снимков проекта.
- Научить студентов уверенно пользоваться working tree, staging area и commit history.
- Отработать ежедневный цикл: inspect changes, stage changes, commit, review log.
- Показать безопасную работу с branches и merge.
- Научить диагностировать и исправлять merge conflicts.
- Сформировать привычку писать понятные commit messages.
- Дать базовые навыки восстановления через restore, revert и reset в безопасных сценариях.

## Модуль 1. Модель данных Git

### Теория

Git хранит историю проекта как направленный граф commits. Каждый commit связан с предыдущим состоянием и содержит снимок файлов, author metadata, timestamp и commit message. Важная идея курса: Git не просто сохраняет файлы, а позволяет объяснить, почему проект пришел к текущему состоянию.

Студенты различают три зоны:

- working tree: текущие файлы на диске;
- staging area: подготовленный набор изменений для следующего commit;
- repository history: уже сохраненные commits.

### Практика

Студенты создают локальный repository, добавляют несколько файлов, выполняют `git status`, `git add`, `git commit`, затем сравнивают состояние working tree и staging area после каждого действия.

### Результаты обучения

- Объяснять различие между working tree, staging area и commit history.
- Определять текущее состояние repository по выводу `git status`.
- Понимать, какие изменения попадут в следующий commit.

## Модуль 2. Осмысленные commits

### Теория

Commit должен описывать законченное изменение. Курс подчеркивает, что плохой commit message мешает code review, debugging и учебной отчетности. Студенты изучают формат короткого imperative message и связь между commit scope и задачей.

Разбираются типичные ошибки:

- слишком большой commit;
- message вида `fix`, `update`, `changes`;
- смешивание refactoring, feature и formatting;
- добавление временных файлов и секретов.

### Практика

Студенты получают папку с учебным проектом и делают серию маленьких commits: исправление README, добавление конфигурации, изменение исходного файла, удаление временного файла. После каждого commit они читают `git log --oneline` и объясняют историю.

### Результаты обучения

- Делить работу на логические commits.
- Писать commit messages, которые объясняют изменение.
- Проверять staged diff перед commit.

## Модуль 3. Просмотр истории и сравнение изменений

### Теория

История Git нужна не только для сохранения, но и для анализа. Студенты изучают `git log`, `git show`, `git diff`, `git diff --staged` и базовые варианты фильтрации. Основной акцент: перед изменением и перед commit нужно понимать diff.

### Практика

Студенты исследуют подготовленный repository с несколькими commits. Нужно найти commit, где была добавлена ошибка в конфигурации, объяснить изменение и предложить исправление отдельным commit.

### Результаты обучения

- Читать diff и отличать добавленные, удаленные и измененные строки.
- Использовать `git show` для объяснения отдельного commit.
- Находить изменение, которое привело к ошибке.

## Модуль 4. Branches как изолированные линии работы

### Теория

Branch в Git — это movable reference на commit. Курс объясняет, что branch не копирует проект, а указывает на точку истории. Студенты изучают `git branch`, `git switch`, `git switch -c`, `git merge`.

Разбираются сценарии:

- отдельная branch для feature;
- отдельная branch для bugfix;
- возврат на main;
- сравнение branch с main.

### Практика

Студенты создают branch `feature/notes-format`, меняют формат заметок, затем возвращаются на `main` и убеждаются, что изменения из branch не видны до merge.

### Результаты обучения

- Создавать и переключать branches.
- Объяснять, почему изменения в feature branch изолированы от main.
- Сравнивать ветку с main перед merge.

## Модуль 5. Merge и конфликты

### Теория

Merge объединяет две линии истории. Если Git не может автоматически совместить изменения, возникает conflict. Курс показывает структуру conflict markers и объясняет, почему конфликт — это не ошибка Git, а сигнал о необходимости человеческого решения.

Студенты изучают:

- fast-forward merge;
- non-fast-forward merge;
- conflict markers;
- manual conflict resolution;
- проверку проекта после разрешения конфликта.

### Практика

В подготовленном repository две ветки меняют одну секцию `README.md`. Студенты выполняют merge, видят конфликт, вручную редактируют файл, запускают проверку и завершают merge commit.

### Результаты обучения

- Распознавать merge conflict.
- Безопасно редактировать файл с conflict markers.
- Завершать merge после разрешения конфликта.

## Модуль 6. Восстановление и безопасная отмена изменений

### Теория

Восстановление в Git требует осторожности. Курс сравнивает `git restore`, `git revert` и `git reset`. Основной принцип: для опубликованной истории безопаснее использовать revert, а reset применять только при понимании последствий.

Студенты изучают:

- отмену unstaged changes;
- снятие файла со staging;
- создание revert commit;
- опасность переписывания shared history.

### Практика

Студенты намеренно портят файл, восстанавливают его через `git restore`, затем создают ошибочный commit и отменяют его через `git revert`.

### Результаты обучения

- Выбирать безопасный способ отмены изменения.
- Объяснять разницу между restore, revert и reset.
- Не использовать destructive reset без осознанной причины.

## Модуль 7. Локальный workflow перед публикацией

### Теория

Перед отправкой изменений в общий репозиторий разработчик должен проверить локальную историю. Этот курс не обучает GitHub pull requests подробно, но готовит основу: clean working tree, понятные commits, актуальная branch, отсутствие секретов.

### Практика

Студенты выполняют checklist перед условной публикацией:

- `git status`;
- `git diff --staged`;
- `git log --oneline --decorate`;
- проверка `.gitignore`;
- проверка отсутствия ключей и временных файлов.

### Результаты обучения

- Проверять локальный repository перед публикацией.
- Находить случайно добавленные секреты и generated files.
- Объяснять, какие commits готовы к отправке.

## Итоговый проект

Студент получает небольшой учебный проект с документацией, конфигурацией и исходными файлами. Нужно:

1. Инициализировать Git repository.
2. Сделать не менее пяти логических commits.
3. Создать feature branch и внести изменение.
4. Смоделировать merge conflict и корректно его разрешить.
5. Отменить один ошибочный commit безопасным способом.
6. Подготовить краткий Markdown-отчет с объяснением истории.

## Критерии успешности

- История содержит логические commits с понятными messages.
- Ветки используются осмысленно.
- Конфликт разрешен без удаления полезного содержания.
- В отчете указано, какие команды применялись и почему.
- В repository нет секретов, временных файлов и лишних generated artifacts.
